# Extraction Instructions (字段级严苛指令)

请严格遵守以下规则对各个字段进行信息提取与推理，严禁主观凭空捏造。

### 1. `meta_routing` (元数据路由)
- `event_category`: 必须基于用户画像，从以下标签中精确选择：["技术发布(Tech)", "开源项目(OpenSource)", "商业融资(Biz)", "人事变动(HR)", "政策监管(Policy)"]。
- `key_entities`: 校验并完善预诊断信息中的实体，格式为 ["模型/项目名", "公司/机构", "关键人名"]。
- `source_tone`: 判定源信息的文体风格，必须从以下四项中精选：“官方发布” / “深度解析” / “大V观点” / “突发快讯”。

### 2. `core_facts` (核心事实锚点)
- `one_sentence_summary`: 严禁使用任何修饰词或最高级（如“极其震撼地发布了”），必须用一句完全客观、无感情色彩的主谓宾句式，阐述发生了什么实质性事件。
- `hard_metrics`: 信息提纯的核心。必须是一个字符串数组。必须包含：具体参数量（如 300B）、上下文窗口长度（如 256k）、具体 Benchmark 跑分（如 MMLU 88%）、算力集群规模（如 10k H100）、API 每百万 Token 准确价格、或融资确切数字。如原文无任何硬核数字，该数组填入 ["无实质性硬核数字表现"]。
- `developer_actionables`: 开发者行动指南。必须提取出可复现、可直接点击或执行的元素，例如：GitHub 源码仓库链接、Hugging Face 权重下载链接、新开放的 API Endpoint 路径、或由于技术升级导致需要立刻替换/废弃的旧技术栈说明。

### 3. `analytical_depth` (深度技术与产业研判)
- `technical_innovation`: 深入挖掘底层。指出该技术在架构、量化、注意力、数据清洗等方面具体的“非微调类”本质创新。如果是纯商业/人事新闻，此处填空。
- `industry_ripple_effect`: 评估蝴蝶效应。该事件对竞争对手或开源/闭源天平会产生什么实质性的博弈影响（例如：该模型的低价发布，是否会迫使闭源厂商调降 API 价格？）。

### 4. `event_fingerprint` (事件指纹)
必须根据提取到的事实生成一个“事件级指纹”对象。用于跨新闻源的精确去重聚类。
- `category_short`: 极短的分类标识 (如 Tech, OpenSource, Biz, HR, Policy)。
- `top_entity`: 核心主语实体 (如 "Llama-3")。
- `claim`: 核心主张或动作极简短语 (如 "发布 8B 权重", "融资 10亿美元")。

### 5. `scoring_alignment` (基础打分与对齐解释)
- `tech_utility_score`: 整数(0-100)。严密对齐用户画像中的【核心关注点】。凡是无开源代码、无 API 更新、无法立刻用于工程落地的纯公关新闻，此分不得超过 40 分。
- `macro_impact_score`: 整数(0-100)。评估该新闻在整个 AI 行业宏观层面的震撼与波及程度。
- `scoring_rationale`: 提供打分理由。
